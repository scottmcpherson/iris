1. Architecture overview

Iris Desktop is a thin client that talks to one backend service: Iris Core. Core is the only component that knows about Hermes — it accesses Hermes through an in-process runtime adapter. The adapter has two channels into Hermes: direct filesystem reads of ~/.hermes, and HTTP calls to the Hermes Gateway and Iris Gateway ports. The reverse channel — Hermes pushing assistant deltas, completions, errors, and cron output back to Iris — is handled by the iris-platform plugin loaded inside Hermes, which posts to POST /v1/runtime-deliveries/hermes.

3. Desktop ↔ Core (HTTP REST + SSE)

4. Desktop Tauri bridge (React → Rust → Python)

In the packaged app, React invokes a Rust Tauri command (core_bridge) that spawns a Python subprocess (apps/desktop/src-tauri/python/core_bridge.py) for Core JSON requests. In Vite/browser development, React uses direct fetch(). The same bridge keeps native-only attachment capabilities: path-based file uploads and attachment byte conversion for WebView playback.

5. Core ↔ Hermes adapter (in-process)

Core does not call any Hermes process directly from its routes. Every route that needs Hermes data resolves a runtime, looks up the adapter in RuntimeRegistry, and calls a method on the adapter object. The single HermesRuntimeAdapter implementation in iris-core/src/hermes_management_server/runtime_adapters/hermes.py then takes one of two paths: filesystem reads/writes against ~/.hermes, or HTTP calls to one of the two Hermes-side ports (Hermes Gateway and Iris Gateway).

6. iris-platform plugin ↔ Core (the reverse channel)

iris-platform is a Hermes platform plugin that runs in-process inside Hermes. On startup it spins up an aiohttp server on {IRIS_INBOUND_HOST}:{IRIS_INBOUND_PORT} exposing the /iris/* routes that the Core adapter calls. Every assistant delta, completion, error, and cron output flows back to Core via a single endpoint: POST /v1/runtime-deliveries/hermes.
